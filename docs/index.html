What this actually is

Not a screenshot filter. A real video driver.

AnsiQuake replaces TyrQuake's SDL video and input backends with a pure termios/ANSI implementation. Every character cell on your terminal displays two vertically-stacked pixels using the Unicode half-block ▀ (U+2580): the foreground color carries the top pixel, the background color carries the bottom — doubling the terminal's effective vertical resolution for free.

One glyph, two pixels. fg = 38;2;R;G;B paints the top pixel, bg = 48;2;R;G;B paints the bottom — a single ▀ per cell reproduces a 2×1 truecolor image with no subpixel trickery, just SGR escape sequences your terminal already understands.

How it works

Three small drivers, zero graphics stack

Everything else — the renderer, physics, netcode, sound mixing — is stock TyrQuake (software-renderer / WinQuake lineage). Only the video and input layers change.

video

common/vid_term.c

Samples the 8-bit palettized software framebuffer down into a (top, bottom) cell grid, diffs against the previous frame, and emits only the cells that changed — with SGR state elision and CSI ?2026 synchronized updates so the terminal never tears. Drops at most every other frame if the tty can't keep pace, so the engine itself never slows down.

input

common/in_term.c

Parses kitty keyboard protocol CSI sequences into Quake's native key press/repeat/release events — the protocol extension that makes WASD movement possible at all, since plain terminal input can't tell you when a key comes back up.

tty

common/term_tty.c

Owns the terminal end-to-end: raw mode, alternate screen, kitty protocol probing, redirecting stdout to console.log, and async-signal-safe restoration on every exit path — fatal signals, SIGTSTP/SIGCONT, crashes, kill. Your shell always comes back clean.

How this got built

One prompt. One autonomous session. A working Quake port.

The entire engine port — plan, three new driver files, upstream patches, and a headless verification harness — came out of a single opening prompt to Claude Code, working unattended from an empty repository to a running build. 

# the opening prompt, verbatim
❯ Produce a new renderer for the game quake. The only catch is it has to use an
  ANSI terminal. Use the ascii half-block to double addressable pixel size.
  This needs to run at original speeds. Use your own judgement when planning
  this, and particularly when spawning subagents. There are no language
  constraints, and you're free to fork from an existing open source project.
  Let's plan this perfectly.

Before you run it

What your terminal needs

  • Truecolor24-bit SGR color support (nearly every modern emulator)
  • Kitty keyboard protocolkitty, ghostty, wezterm, foot, or alacritty ≥ 0.13
  • A real ttytmux and screen sessions aren't supported — the protocol doesn't pass through
  • ≥ 320×200 pixel gridcols × 2·rows; below that the renderer downsamples and text gets garbled

Try it yourself

Up and running in four commands

# needs libsdl2-dev (sound only — video is never touched)
$ git clone https://github.com/pixelbadger/AnsiQuake.git
$ cd AnsiQuake
$ make terminal              # builds bin/tyr-quake
$ scripts/fetch-shareware.sh  # pulls id1/pak0.pak (shareware ep.)
$ ./bin/tyr-quake

Resizing mid-game works. Quit from the menu, or hit Ctrl+C three times within a second for an emergency exit — the terminal is restored no matter how the process dies.
